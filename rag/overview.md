# RAG 总览

检索增强生成（RAG）通过外部知识检索提升 LLM 的事实准确性和时效性，在不重训模型的前提下引入可控的知识来源。

## 核心流程

```
查询 → 检索模块 → 上下文构建 → LLM生成
         │
    ├─ Embedding模型
    ├─ 向量存储
    ├─ 混合检索
    └─ 重排序
```

## 技术分类

RAG 系统可以从多个维度分类：

| 分类维度 | 类型 | 说明 |
|---------|------|------|
| 检索粒度 | 文本块级 / 实体级 / 段落级 | 粗粒度检索覆盖广，细粒度检索精度高 |
| 检索方式 | 向量检索 / 图检索 / 混合检索 | 向量语义匹配，图结构化查询，混合兼顾两者 |
| 交互模式 | 单轮 / 多轮 / 迭代 | 单轮简单直接，多轮支持追问，迭代可纠错 |
| 复杂度 | Naive RAG / Advanced RAG / Modular RAG | 从简单拼接检索结果到模块化编排 |
| 知识注入 | 静态索引 / 增量更新 / 在线学习 | 静态简单，增量可持续，在线学习动态适应 |

## 能力边界

**RAG 擅长：**
- 需要引用外部知识源的问答（文档、数据库、知识库）
- 频繁更新、无法通过重训练覆盖的知识场景
- 多跳推理（配合知识图谱或迭代检索）
- 领域定制（特定领域文档作为检索源）

**RAG 不擅长：**
- 需要深度推理链的场景（此时 CoT 或 Agent 更合适）
- 答案形式高度固定的分类/抽取任务（微调更高效）
- 检索源质量差时的幻觉（garbage in, garbage out）
- 低延迟要求场景（检索 + 生成的双重开销）

## 关键 trade-off

| 取舍 | 说明 |
|------|------|
| 检索召回 vs 精度 | 召回高引入噪声，精度高可能遗漏相关信息 |
| Chunk 大小 | 大 chunk 上下文丰富但检索粗粒度，小 chunk 精准但丢失关联 |
| 检索深度 | 多跳/多轮检索提高覆盖率，但增加延迟和错误累积 |
| 向量 vs 图 | 向量检索通用但缺结构，图检索精确但构建成本高 |
| 端到端 vs 模块化 | 端到端优化一致但解释性差，模块化灵活但调试复杂 |
