# Programming with Data: Test-Driven Data Engineering for Self-Improving LLMs from Raw Corpora

> **一句话总结**：将软件工程的「测试驱动开发」范式迁移到 LLM 数据工程中，通过结构化知识表示实现训练数据与模型行为的可追溯、可修复对应关系。

## 基本信息

- **论文**: Programming with Data: Test-Driven Data Engineering for Self-Improving LLMs from Raw Corpora
- **作者**: Chenkai Pan, Xinglong Xu, Yuhang Xu, Yujun Wu, Siyuan Li, Jintao Chen, Conghui He, Jingxuan Wei, Cheng Tan
- **机构**: Zhejiang University, University of Chinese Academy of Sciences, Shanghai Artificial Intelligence Laboratory
- **发表**: arXiv, 27 Apr 2026
- **链接**: https://arxiv.org/abs/2604.24819v1
- **arXiv ID**: 2604.24819

## 1. 问题与动机

### 1.1 核心问题

将专业领域知识从文本可靠地迁移到大语言模型中，是人工智能的一个根本性挑战。当前的领域微调（fine-tuning）存在**反馈缺失**问题：

- 当模型在领域任务上失败时，**无法诊断**训练数据的缺陷
- 唯一的补救措施是**无差别地增加更多数据**
- 数据 engineering 过程是**黑盒**，训练数据与模型行为之间没有可追溯的映射

### 1.2 为什么重要

- 领域知识迁移是 LLM 实际应用的核心瓶颈（医疗、法律、科学等）
- 现有方法依赖「越多越好」的直觉，缺乏工程化的质量控制
- 如果数据缺陷不可诊断，就无法系统性地提升模型在特定领域的能力

## 2. 背景

### 2.1 技术演进脉络

| 阶段 | 代表工作 | 核心思路 | 局限 |
|------|----------|----------|------|
| 预训练 + 提示 | GPT-3, ChatGPT | 通用知识 + 上下文学习 | 领域深度不足 |
| 领域微调 | LoRA, QLoRA | 在领域语料上微调 | 无反馈机制，黑盒优化 |
| 数据合成 | Self-Instruct, Alpaca | 用 LLM 生成训练数据 | 质量不可控，合成偏差 |
| **本文** | Programming with Data | 结构化知识 + 测试驱动 | — |

### 2.2 与软件工程的类比灵感

软件工程从「写代码→编译→测试→调试」的闭环中获得了可靠性。本文的核心洞察是：**数据工程可以采用同样的闭环结构**，前提是训练数据和评估共享同一个结构化基础。

## 3. 方法

### 3.1 核心范式：Programming with Data (ProDa)

论文形式化了一个四元对应关系：

| 软件工程 | 数据工程（本文） | 说明 |
|----------|-----------------|------|
| **源代码** | **训练数据** | 指定模型应该学习什么 |
| **编译** | **模型训练** | 将规范转换为可执行模型 |
| **单元测试** | **基准测试** | 验证模型是否满足规范 |
| **调试** | **失败驱动的数据修复** | 定位缺陷并针对性修复 |

### 3.2 关键设计：结构化知识表示

**为什么需要结构化知识？**

- 原始语料是**非结构化的文本**，无法直接用于诊断模型失败
- 结构化知识表示（从源语料提取）作为**训练和评估的共享基础**
- 模型失败可以分解为：
  - **概念级缺口**（concept-level gaps）：某些概念在训练数据中覆盖不足
  - **推理链断裂**（reasoning-chain breaks）：多步推理中的某个环节缺失

### 3.3 ProDa 管线

```
原始语料
    ↓
结构化知识提取（Specification）
    ↓
┌─────────────────┬─────────────────┐
↓                 ↓                 ↓
训练数据合成    基准测试合成      失败诊断
（基于知识）    （基于知识）      （基于知识）
    ↓                 ↓                 ↓
模型训练      ←  基准测试评估  ←  定位数据缺陷
    ↓                 ↓                 ↓
模型能力    ←  测试通过/失败  ←  针对性修复数据
```

### 3.4 关键设计决策

**决策 1：结构化知识作为共享基础**
- **原因**：只有训练和评估基于同一套结构化表示，失败才能追溯到数据缺陷
- **替代方案**：直接用原始文本作为训练数据 → 无法诊断，因为失败与数据之间没有显式映射

**决策 2：失败分解为概念缺口和推理链断裂**
- **原因**：这两种失败类型对应不同的数据修复策略
  - 概念缺口 → 补充缺失概念的训练样本
  - 推理链断裂 → 补充中间推理步骤的训练样本
- **替代方案**：笼统地「增加更多数据」→ 无法针对性修复，效率低

## 4. 实验

### 4.1 实验设置

- **领域覆盖**：16 个学科（自然科学、工程、生物医学、社会科学）
- **模型规模**：跨多个模型规模和架构验证
- **基准**：ProDa-16 基准套件 + MMLU/C-Eval 相关子集

### 4.2 主要结果

| 指标 | 结果 |
|------|------|
| 学科覆盖率 | 16 个学科 |
| 知识图谱连通性 | 所有学科 > 99%，11/16 学科 > 99.8% |
| 修复效果 | 每次修复周期在跨模型规模和架构上产生一致改进 |
| 通用能力 | 修复过程不降低通用能力 |

### 4.3 关键发现

- **结构化知识的价值**：知识图谱的高连通性（>99%）证明了从原始语料提取结构化知识的可行性
- **可追溯性**：模型失败可以精确追溯到训练数据中的特定缺陷
- **修复效率**：针对性修复比盲目增加数据更有效

## 5. 分析与讨论

### 5.1 关键洞察

**洞察 1：数据工程可以工程化**

本文最大的贡献是证明了数据工程不必是「炼丹术」。通过引入结构化知识和测试驱动反馈，数据工程可以像软件工程一样系统化、可诊断、可修复。

**洞察 2：训练数据与模型行为的对应关系是可形式化的**

传统观念认为训练数据与模型行为的关系过于复杂，无法分析。本文通过结构化知识表示，证明了这种对应关系在**概念级别**和**推理链级别**是可以追溯的。

**洞察 3：失败是信息，不是终点**

在传统的数据工程流程中，模型失败是一个「终点」——意味着需要更多数据。在 ProDa 中，失败是一个「信号」——指向数据中的具体缺陷，驱动有针对性的修复。

### 5.2 局限性（含根因分析）

| 局限 | 根因分析 |
|------|----------|
| 结构化知识提取的质量依赖源语料质量 | 如果原始语料本身不完整或有偏差，提取的知识结构也会有偏差。这是所有基于语料的方法的共同局限，但 ProDa 至少可以通过测试暴露这些偏差。 |
| 16 个学科的覆盖仍有限 | 领域知识提取需要领域专家验证，大规模扩展需要自动化验证机制。本文发布了 ProDa Studio 作为半自动化的 IDE，但完全自动化仍是挑战。 |
| 失败诊断的粒度受限 | 概念级和推理链级别的诊断依赖于知识图谱的结构。对于更细粒度的问题（如特定数值计算错误），可能需要额外的诊断机制。 |
| 修复策略的通用性 | 本文主要关注「补充缺失数据」的修复策略。对于「修正错误数据」（如语料中的事实错误）的修复，需要额外的验证机制。 |

### 5.3 未来方向

1. **自动化知识提取**：当前仍需要一定程度的人工干预来验证提取的知识结构，完全自动化的知识提取是下一步
2. **动态知识更新**：领域知识在演进，如何实现知识结构的增量更新而不破坏已有训练
3. **多模态扩展**：当前主要针对文本语料，图像、表格等多模态数据的结构化表示是自然的延伸
4. **开源生态**：ProDa Studio 作为 IDE 的成熟度和社区 adoption 将决定这一范式的实际影响

## 6. 结论

本文提出了 **Programming with Data** 范式，将软件工程的测试驱动开发思想系统性地迁移到 LLM 数据工程中。核心贡献在于：

1. **形式化了训练数据与模型行为的对应关系**——通过结构化知识表示，证明这种关系是可追溯、可修复的
2. **建立了数据工程的闭环**——从训练数据合成 → 模型训练 → 基准测试 → 失败诊断 → 数据修复
3. **在 16 个学科中验证了可行性**——证明了这一范式在不同领域中的通用性

这一工作为「如何可靠地将人类专业知识工程化到语言模型中」提供了原则性的基础，可能标志着数据工程从「炼丹术」向「工程学」的转变。
