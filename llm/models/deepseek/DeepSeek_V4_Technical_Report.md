## 一、论文整体思路

### 1.1 研究背景：长上下文的效率瓶颈

推理模型的出现构建了测试时间缩放（Test-Time Scaling）的全新范式，为大语言模型带来了显著的性能提升。然而，这种缩放范式从根本上受到基础注意力机制**二次计算复杂度**的限制——序列长度增加 $N$ 倍，注意力计算量增长 $N^2$ 倍，KV缓存线性膨胀。这为超长上下文和推理过程造成了难以承受的瓶颈。

与此同时，长跨度场景与任务的兴起——从复杂的智能体工作流到大规模跨文档分析——也使得对超长上下文的高效支持成为关键需求。尽管近期开源领域的相关努力推动了通用能力的提升，但在处理超长序列时，这一核心架构效率问题仍是主要阻碍，它不仅限制了测试时间缩放带来的进一步性能提升，也阻碍了对长跨度场景与任务的进一步探索。

传统Transformer处理长文本的死穴可以归结为：

| 问题 | 具体表现 | 量级 |
|------|----------|------|
| 计算量平方级爆炸 | 每个token需与历史所有token计算相似度 | 序列长度N → 计算量O(N²) |
| KV缓存线性膨胀 | 每个token的KV状态需持久存储 | 128K上下文 → 数十GB显存 |
| 长程依赖衰减 | 注意力权重在远距离token上趋于均匀 | 超过32K后检索精度骤降 |

### 1.2 DeepSeek-V4的定位与核心贡献

DeepSeek-V4系列的定位非常明确：**突破超长上下文的效率瓶颈，使百万token上下文从"奢侈品"变为"日用品"**。系列包含两款MoE模型：

| 模型 | 总参数 | 激活参数 | 上下文长度 | 定位 |
|------|--------|----------|------------|------|
| DeepSeek-V4-Pro | 1.6T | 49B | 1M | 旗舰，对标顶级闭源模型 |
| DeepSeek-V4-Flash | 284B | 13B | 1M | 高效，极致性价比 |

核心贡献可归纳为三大架构创新：

1. **混合压缩注意力架构**：融合压缩稀疏注意力（CSA）与重度压缩注意力（HCA），从根源上改造注意力机制，实现"压缩+稀疏化+混合精度+滑动窗口+注意力沉降"多管齐下
2. **流形约束超连接（mHC）**：通过Birkhoff多面体投影约束残差连接的混合矩阵，解决万亿参数训练中的信号爆炸/消失问题
3. **Muon优化器**：用矩阵级正交化替代逐参数调整，实现更快收敛与更稳定训练

在100万token上下文场景下，DeepSeek-V4-Pro的单token推理FLOPs仅为V3.2的**27%**，KV缓存仅为其**10%**；V4-Flash更为极致，分别压到**10%**和**7%**。

## 二、模型架构总览

### 2.1 架构设计哲学：从参数稀疏到上下文稀疏

DeepSeek的技术路线存在一条清晰的哲学主线：

```
V2 → 参数稀疏化（MoE：总参数大，每token只激活一小部分专家）
V3 → 参数稀疏化深化（更高效的MoE + 无辅助损失的负载均衡）
V4 → 参数稀疏化 + 上下文稀疏化（MoE + 压缩注意力：不仅参数稀疏，KV缓存和注意力计算也稀疏）
```

V4的突破在于，在参数稀疏化（MoE）的基础上，**新开辟了上下文稀疏化这条路线**——KV压缩、Top-k选择、分层压缩率协同工作。这是DeepSeek第一次把稀疏化的思路推进到Transformer的核心结构（注意力机制）里。

设计哲学的转变可以用一个类比来理解：
- **传统注意力**：像要求一个人把整本书一字不差地背下来
- **V4混合注意力**：把"记忆"整理成层次化的摘要——有些是每一小段的凝练（CSA），有些是每一章的概括（HCA），再加上眼前几句话的原文（滑动窗口）。需要调用哪一层、哪一段，由模型自己根据当前问题即场判断

## 三、混合压缩注意力机制

### 3.1 传统注意力的长上下文困境

标准Transformer注意力的计算流程为：

$$\text{Attention}(Q, K, V) = \text{softmax}\left(\frac{QK^T}{\sqrt{d_k}}\right)V$$

当序列长度为 $N$ 时：
- **计算复杂度**：$O(N^2 \cdot d)$，每个query需与所有历史key计算点积
- **KV缓存**：$O(N \cdot d)$，每个token的KV状态需持久存储
- **二次增长**：上下文从10K扩展到1M，计算量增长10000倍

之前的MLA（多头潜在注意力）通过将KV压缩到低维潜在空间来减少缓存，但本质上仍是全序列的密集注意力，未从根本上解决二次计算复杂度问题。

### 3.2 压缩稀疏注意力（CSA）

CSA是V4注意力架构的核心组件，其设计思想是"先压缩、再选择、后计算"。数据流分为三路并行处理。

#### 3.2.1 Token级压缩器

KV token的隐藏状态首先进入Token-Level Compressor，将每 $m$ 个原始token压缩成一个压缩KV条目。CSA默认 $m=4$，即序列长度直接变为原来的四分之一。

压缩过程**不是简单平均**，而是学习式加权压缩：

$$\text{CompressedKV}_i = \sum_{j=1}^{m} \alpha_{ij} \cdot \text{KV}_{ij} + \beta_i$$

其中：
- $\alpha_{ij}$：每个原始token的学习权重，通过位置偏置（Positional Bias）和softmax归一化获得
- $\beta_i$：位置偏置项
- **相邻压缩块重叠**：避免边界信息丢失，确保跨块依赖不因压缩而断裂

这种"学出来的压缩"保证了信息保留的关键特性——哪些信息重要、哪些可以丢弃，由模型在训练中自主决定。

#### 3.2.2 Lightning Indexer与稀疏选择

压缩后并非对所有压缩条目计算注意力，而是通过Lightning Indexer进行**学出来的动态筛选**：

**查询端**：
1. Query隐藏状态降维到 $d_c = 1024$（减少参数量）
2. 升维到多头索引查询
3. 经ReLU激活后与压缩索引键计算相关性分数

**键端**：
1. KV隐藏状态同时进入Lightning Indexer
2. 生成"索引键"（Index Key）用于后续匹配打分
3. 生成"索引分数"（Index Score）

**Top-k选择**：
- 两端分数合并后，送入Top-k选择器
- **Pro版**：从所有压缩块中选出最相关的 **1024** 个
- **Flash版**：选出最相关的 **512** 个

最终，选中的压缩块与滑动窗口保留的128个原始token拼接，送入核心的多查询注意力（MQA）计算。

Lightning Indexer的设计哲学：**不是简单截断，而是学出来的动态筛选**。这保证了在不同上下文和任务中，模型能自适应地选择最相关的信息。

#### 3.2.3 滑动窗口与注意力沉降

**滑动窗口**：保留最近128个token的原始KV不压缩，确保局部依赖没有精度损失。最近的token往往是最相关的，这一设计兼顾了全局和局部细节。

**注意力沉降（Attention Sink）**：给每个注意力头一个**可学习的sink logit**，加到注意力分数的分母里。这让每个头可以选择"什么都不关注"，避免注意力被强制分配到不相关的远距离token上。

CSA完整数据流：

```
KV Token隐藏状态 ──→ Token-Level Compressor (m=4) ──→ 压缩KV条目
                    ──→ Lightning Indexer ──→ 索引键 + 索引分数
                                               ↘
Query隐藏状态 ──→ Lightning Indexer ──→ 查询索引分数 ──→ Top-k选择
                                                        ↓
                                              选中的压缩KV + 滑动窗口(128)
                                                        ↓
                                                  MQA注意力计算
                                                        ↓
                                                    注意力输出
```

### 3.3 重度压缩注意力（HCA）

HCA与CSA形成互补，其设计哲学截然不同——**去掉稀疏选择链路，用极端压缩换取全量注意力**。

| 对比维度 | CSA | HCA |
|----------|-----|-----|
| 压缩比 $m'$ | 4 | 128 |
| Lightning Indexer | 有 | 无 |
| Top-k选择 | 有（512/1024） | 无 |
| 注意力类型 | 稀疏MQA | 全量MQA |
| 目标 | 中段关联的精准检索 | 全局结构的粗粒度感知 |

HCA的数据流非常直接：
1. KV隐藏状态经过Token-Level Compressor，以 $m'=128$ 的压缩比生成压缩条目
2. 压缩条目直接与滑动窗口KV拼接
3. 送入MQA进行全量密集注意力计算

因为每条目覆盖128个原始token，条目总量已经极少，全量计算也不昂贵，省掉了筛选环节。HCA的目标是**抓全局结构**——让模型"既见树木又见森林"，避免CSA的稀疏选择遗漏全局脉络。

### 3.4 CSA与HCA的交错配置策略

CSA和HCA并非简单并列，而是在模型层间**交替叠用**：

- **CSA层**：负责中距离的精准关联检索（"这一段和哪几段相关？"）
- **HCA层**：负责全局结构的粗粒度感知（"整篇文章的主题脉络是什么？"）
- **滑动窗口**：两层共享，负责近距离的精确依赖（"最近几个token之间的关系"）

这种"粗粒度+细粒度、稀疏+稠密"的组合方案，使得局部细节、中段关联、全局脉络全都被抓住，而算力开销断崖式下降。

### 3.5 位置编码与混合精度KV缓存

**位置编码策略**：CSA和HCA在Query和KV上**仅对最后64个维度施加RoPE位置编码**，其余维度不编码。这一工程决定既保留了位置感知能力，又避免了RoPE在压缩KV上的不一致问题。

**混合精度KV缓存**：KV缓存采用混合精度存储：
- RoPE维度：**BF16**（保证位置编码精度）
- 非RoPE维度：**FP8**（大幅压缩缓存体积）

这一策略将KV缓存体积再砍近一半，与压缩机制叠加实现极致的缓存效率。

### 3.6 异构KV缓存管理

混合注意力有多种不同类型的KV，压缩比和更新规则各不相同，传统分页缓存（PagedAttention）无法直接适配。V4将KV缓存重新设计为两大块：

**1. 状态缓存（State Cache）**
- 存储内容：滑动窗口最近128个token的KV + CSA/HCA中尚未攒够压缩数量（4或128）的"尾料"token
- 分配方式：每个请求预分配固定大小的块
- 特点：高频访问，常驻GPU显存

**2. 经典压缩缓存（Classical Compressed Cache）**
- 存储内容：已经压缩好的CSA和HCA的KV条目
- 对齐策略：每个块覆盖的原始token数为两种压缩比的最小公倍数 $\text{LCM}(4, 128) = 512$
  - CSA产 $512/4 = 32$ 个压缩条目
  - HCA产 $512/128 = 1$ 个压缩条目
  - 同一块内CSA和HCA的压缩结果可以对齐，避免碎片化
- 磁盘缓存：支持共享前缀的磁盘存储，重复请求不用重新预计算

这套缓存布局是百万上下文从实验室走进生产环境的关键工程基础设施。

### 3.7 效率提升量化分析

V4在100万token上下文下的效率提升（基准：DeepSeek-V3.2）：

| 指标 | V4-Pro | V4-Flash |
|------|--------|----------|
| 单token推理FLOPs（等效FP8） | **27%** | **10%** |
| KV缓存大小 | **10%** | **7%** |

需要强调的是，单token推理算力的单位是**等效FP8 FLOPs**，不是拿低精度取巧，而是换算到与V3.2相同的精度标准去比较。而且，V4系列的路由专家权重还用了FP4精度——报告特别指出，现有硬件上FP4和FP8的峰值算力相同，但**未来硬件上FP4可以再高出1/3的效率**。

## 四、流形约束超连接（mHC）

### 4.1 残差连接的演进与深层网络信号问题

残差连接（Residual Connection）是深度学习的基石，其标准形式为：

$$\mathbf{y} = \mathbf{x} + f(\mathbf{x})$$

超连接（Hyper-Connections, HC）对残差连接进行了扩展，将残差流的宽度放大，不增加内层计算量就能提升模型表达能力。混合形式为：

$$\mathbf{y} = \alpha \cdot \mathbf{x} + \beta \cdot f(\mathbf{x})$$

其中 $\alpha, \beta$ 为可学习的混合系数。但HC存在致命问题：**堆叠多层后训练极度不稳定**。根本原因在于，信号在穿过数十层时，每层的混合系数都可能放大信号——逐层累积后，信号放大可达 $10^3$ 到 $10^5$ 倍，导致梯度爆炸或数值溢出。

### 4.2 mHC的数学约束：Birkhoff多面体投影

mHC的核心思想是给残差连接加一个**数学笼子**——强制每层的混合矩阵满足双随机矩阵（Doubly Stochastic Matrix）约束：

$$W \in \mathcal{B}_n = \{W \in \mathbb{R}^{n \times n}_{\geq 0} : W\mathbf{1} = \mathbf{1}, W^T\mathbf{1} = \mathbf{1}\}$$

即混合矩阵的**每行元素之和为1**、**每列元素之和为1**、**所有元素非负**。这个约束集合 $\mathcal{B}_n$ 就是Birkhoff多面体——所有双随机矩阵构成的空间。

直观理解：双随机矩阵保证了无论信号如何传播，**幅度不会发散也不会衰减**。每个输出维度从各输入维度获得的"信号量"总和恒为1，如同一个完美的信号分配器。

### 4.3 Sinkhorn-Knopp迭代实现

将任意矩阵投影到Birkhoff多面体上的标准方法是Sinkhorn-Knopp算法，即交替进行行归一化和列归一化：

```
输入：原始混合矩阵 W
for i = 1 to 20:
    W = W / W.sum(dim=1, keepdim=True)   # 行归一化
    W = W / W.sum(dim=0, keepdim=True)   # 列归一化
输出：投影后的双随机矩阵 W
```

DeepSeek-V4采用**20次**交替行列归一化，硬把矩阵拉回约束集合。实现简洁、数学上干净、可微（梯度可回传）。

### 4.4 信号放大抑制效果

mHC的约束效果显著：

| 方法 | 信号放大倍数 | 训练稳定性 |
|------|-------------|-----------|
| 标准残差连接 | 1x（恒等） | 稳定但表达力有限 |
| 无约束HC | $10^3 \sim 10^5$x | 极不稳定 |
| **mHC** | **< 2x** | **稳定** |

训练开销仅增加**6.7%**，换来了万亿参数模型训练的稳定性。这是典型的"用少量计算换大量安全"的设计。

基准验证（BBH指标）：
- 标准残差连接：43.8
- 无约束HC：48.9（训练不稳定）
- mHC：**51.0**（训练稳定 + 更高性能）

## 五、Muon优化器与训练稳定性

### 5.1 从AdamW到Muon：正交化梯度更新

传统优化器（如AdamW）逐参数调整更新方向，本质上是对每个参数独立做自适应学习率。Muon则采取完全不同的策略——**矩阵级正交化**。

Muon的核心步骤：
1. 获取参数矩阵的梯度 $G$
2. 对梯度矩阵进行**Newton-Schulz迭代**（牛顿-舒尔茨迭代），将其正交化
3. 正交化后的梯度各行更新方向相互独立、不冲突
4. 每次更新踩在最干净的方向上，同样步数学到更多

效果：**变相省算力**——在相同计算预算下，Muon比AdamW收敛更快、训练更稳定。

V4中大部分参数使用Muon，仅嵌入层（Embedding）、输出头（Output Head）和RMSNorm仍用AdamW。

### 5.2 配合Muon的分布式策略调整

Muon的矩阵级操作对分布式训练策略提出了新要求，V4做了针对性调整：

**稠密参数**：限制切分粒度，每个GPU最多管5个完整矩阵（避免矩阵被切碎后无法做正交化）

**MoE参数**：直接拼成大向量等分，不切单个矩阵（保证专家内部的矩阵完整性）

**梯度通信**：做BF16量化，砍掉一半通信量（矩阵级正交化后梯度分布更规整，BF16量化损失更小）

### 5.3 训练稳定性技巧

万亿参数训练不可避免地遇到损失尖峰（Loss Spike），V4用两个工程技巧解决：

#### 5.3.1 预判路由（Predictive Routing）

**问题**：路由网络和骨干网络同步更新会形成正反馈循环——路由偏向某些专家 → 这些专家更新更快 → 路由更偏向它们 → 负载不均加剧 → Loss Spike

**解法**：路由用历史参数（稍旧的版本）算索引，提前缓存路由结果，打破同步更新循环

**触发策略**：仅在检测到Loss Spike时自动开启，平时不增加额外开销

#### 5.3.2 SwiGLU截断

**问题**：SwiGLU激活函数在某些输入下产生极大的数值outliers，导致梯度爆炸

**解法**：将SwiGLU的线性分量限制在 $[-10, 10]$，门分量上限10

**效果**：消除数值outliers，稳定训练，几乎不影响正常范围内的表达能力

### 6.2 序列长度渐进策略

预训练采用序列长度逐步拉长的策略：

```
4K → 32K → 128K → 1M
```

这种渐进策略的工程考量：
- 短序列阶段用大批量快速学习语言模式
- 长序列阶段用小批量但充分学习长程依赖
- 避免一开始就训练1M序列导致的显存溢出和计算浪费

预训练完成后，两个模型能够**原生且高效**地支持100万长度的上下文，无需额外的上下文扩展后处理。

- **路由专家**：参数采用FP4精度（量化感知训练）
- **共享专家**：保留V3的Shared Expert设计
- **负载均衡**：沿用V3的无辅助损失（Auxiliary-Loss-Free）负载均衡策略
- **专家路由**：预判路由机制应对Loss Spike

### 6.4 多令牌预测（MTP）

MTP配置与V3保持一致。每个MTP模块在主预测头之外，额外预测1-2个未来token，作为训练辅助目标。MTP的作用：
- 提供更密集的训练信号
- 增强模型的规划能力
- 不增加推理成本（推理时可选择是否使用MTP头）

## 七、后训练流程

### 7.1 两阶段范式：先分后合

V4的后训练流程采用了全新的两阶段范式，与传统"混合训练"方式有本质区别：

```
阶段一：分头训专才
┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐
│ 数学专家  │ │ 编程专家  │ │ 智能体专家│ │ 指令专家  │
│ SFT→GRPO │ │ SFT→GRPO │ │ SFT→GRPO │ │ SFT→GRPO │
└──────────┘ └──────────┘ └──────────┘ └──────────┘

阶段二：无损蒸馏合体
┌──────────────────────────────────────────────────┐
│              统一模型（学生）                       │
│     学习优化与所有教师之间的反向KL散度               │
└──────────────────────────────────────────────────┘
```

### 7.2 领域专家独立训练

每个领域专家的训练流程为：

**第一步：有监督微调（SFT）**
- 在高质量的领域专属数据上进行SFT
- 建立基础能力

**第二步：强化学习（GRPO）**
- 采用分组相对策略优化（GRPO）算法
- 配合领域专属的奖励模型
- 奖励模型本身也是**生成式的**——让模型同时学会"判卷"和"答卷"，减少对人类标注的依赖

各领域专家的特点：
- 数学专家：侧重推理链验证、步骤正确性
- 编程专家：侧重代码执行结果、测试通过率
- 智能体专家：侧重任务完成率、工具使用正确性
- 指令遵循专家：侧重格式遵从、约束满足

### 7.3 在线策略蒸馏与反向KL散度

将多个专家的能力融合到统一模型中，V4采用**在线策略蒸馏（Online Policy Distillation）**：

**核心方法**：
- 统一模型作为学生，学习优化与教师模型之间的**反向KL散度（Reverse KL Divergence）**
- 不是token级近似，而是**全词表级别**（128K词表）的logit分布匹配
- 保持教师完整的logit分布，确保知识无损传递

**工程挑战与解法**：

| 挑战 | 解法 |
|------|------|
| 词表128K × 十多个万亿参数教师 = 巨大计算压力 | 教师权重从中心化存储按需加载 |
| 存储完整logits显存爆炸 | 不存完整logits，只缓存最后一层隐藏状态，训练时即时重算 |
| 多教师同时加载显存不足 | 按教师索引排序样本，保证同一时刻GPU上只有一个教师头 |

这种方法比传统的混合RL效果更好，不会出现不同领域之间的性能退化。

### 7.4 三档推理模式

V4为不同场景设计了三档推理模式，在RL训练时分别用不同的上下文窗口和惩罚系数：

| 模式 | 触发方式 | 特点 | 适用场景 |
|------|----------|------|----------|
| Non-think | 无思考标签 | 快速回答，无思维链 | 日常快速响应 |
| Think High | 显式思维链 | 受控长度，平衡深度和速度 | 复杂问题 |
| Think Max | 特殊系统提示 + 放宽长度惩罚 | 极限思维模式，推理能力拉满 | 冲推理极限 |

三种模式共享同一套模型权重，通过不同的推理配置切换深度。

此外，工具调用也改进了新的schema，降低调用错误率，同时支持保留多轮推理历史，长周期Agent任务不用每次重新构建状态。

## 八、基础设施与系统优化

### 8.1 MoE融合内核与通信-计算重叠

为MoE模块设计并实现了**单一融合内核（Single Fused Kernel）**，可实现：
- **计算**（专家前向/反向）
- **通信**（All-to-All专家并行通信）
- **内存访问**（专家权重加载/存储）

三者的**完全重叠**。在等待通信完成的同时，已经在计算其他专家的结果或预取下一批权重，最大化硬件利用率。

### 8.2 TileLang DSL

采用领域特定语言（DSL）**TileLang**，在开发效率与运行时效率之间取得平衡：
- 用高层抽象描述算子逻辑，提升开发效率
- 编译到底层高效实现，保证运行时性能
- 对MoE、混合注意力等自定义算子尤为重要——这些算子无法直接用标准CUDA内核实现

### 8.3 FP4量化感知训练

针对MoE专家权重和索引器的QK路径引入**FP4量化感知训练（QAT）**：

- **MoE专家权重**：FP4精度
- **QK路径**：FP4精度
- **其他参数**：FP8精度
- **当前硬件**：FP4×FP8运算的峰值FLOPs与FP8×FP8持平
- **未来硬件**：理论上可实现1/3的效率提升

这一策略在不损失模型性能的前提下，显著降低了内存占用和计算量。

### 8.4 张量级检查点与混合ZeRO策略

**张量级检查点**：为autograd框架扩展了张量级检查点功能，实现细粒度的重计算控制。相比层级检查点，可以选择性地重计算某些张量而非整层，在显存和计算之间取得更优平衡。

**混合ZeRO策略**：
- 适配Muon优化器的矩阵级操作
- 稠密参数：限制切分粒度
- MoE参数：大向量等分
- 通过重计算与融合内核实现高性价比的混合精度计算

### 8.5 两阶段上下文并行

采用两阶段上下文并行机制处理压缩注意力：
- **第一阶段**：序列维度的上下文并行（处理长序列的分片）
- **第二阶段**：压缩注意力的特殊并行（处理CSA/HCA的跨分片KV访问）

这是支持1M上下文训练的关键系统优化。

### 8.6 推理框架：异构KV缓存与磁盘存储

推理框架的核心创新是**异构KV缓存结构**：

```
┌──────────────────────────────────────────────┐
│           GPU 显存（高速、小容量）              │
│  ┌──────────────────────────────────────┐    │
│  │  状态缓存：滑动窗口KV + 未压缩尾料     │    │
│  └──────────────────────────────────────┘    │
├──────────────────────────────────────────────┤
│           CPU 内存（中速、中容量）              │
│  ┌──────────────────────────────────────┐    │
│  │  经典压缩缓存：CSA/HCA压缩KV条目      │    │
│  └──────────────────────────────────────┘    │
├──────────────────────────────────────────────┤
│           磁盘存储（低速、大容量）              │
│  ┌──────────────────────────────────────┐    │
│  │  共享前缀缓存：重复请求直接复用        │    │
│  └──────────────────────────────────────┘    │
└──────────────────────────────────────────────┘
```

不同策略适配不同场景，平衡存储和计算。共享前缀的磁盘缓存特别适合多请求复用同一系统提示的场景。

## 十一、关键见解与总结

### 核心贡献

1. **混合压缩注意力是V4最根本的架构创新**：CSA负责中距离精准检索，HCA负责全局结构感知，滑动窗口保证局部精度。三者协同实现了"压缩+稀疏化+混合精度+滑动窗口+注意力沉降"多管齐下的效率革命
2. **mHC解决了万亿参数训练的稳定性难题**：通过Birkhoff多面体投影将信号放大从$10^3$-$10^5$倍压到<2倍，仅6.7%的额外开销
3. **Muon优化器带来更快的收敛**：矩阵级正交化让每次更新更高效，配合专门设计的分布式策略
4. **后训练的"先分后合"范式**：独立训练领域专家 → 在线策略蒸馏融合，反向KL散度保证知识无损传递
5. **FP4量化感知训练**：在不损失性能的前提下进一步压缩MoE专家权重，为未来硬件预留效率空间

### 关键见解

- **效率是V4的核心命题**：27%的FLOPs和10%的KV缓存不是渐进改进，而是架构级别的范式转换
- **稀疏化的思路已从参数扩展到注意力机制的核心**：V4是DeepSeek第一次把稀疏化推进到Transformer的核心结构
- **"学出来的压缩"优于"设计出来的压缩"**：Token-Level Compressor和Lightning Indexer都是端到端可训练的，模型自己决定哪些信息重要
- **工程细节决定架构创新能否落地**：异构KV缓存、LCM对齐、预判路由、SwiGLU截断等工程技巧，是百万上下文从论文走向生产的关键
- **国产算力适配是战略级布局**：不仅是对制裁的应对，更是对未来更大规模推理部署的前瞻准备
