# domleca/llm-wiki 深度分析

> **一句话总结**：将 Karpathy LLM Wiki 理念编译为 Obsidian 生态内最小摩擦、隐私优先的本地 AI 知识引擎。

## 项目定位

### 解决的问题
Obsidian 用户面临“信息过载但知识匮乏”的困境——笔记越积越多，但笔记之间的隐性关联和知识结构无法被传统的关键词搜索或标签系统发现。LLM Wiki 插件通过自动实体/概念提取、关系构建和混合搜索，把碎片化的笔记转化为**可对话、可导航的结构化知识库**。

### 目标用户
- 已有成熟 Obsidian 工作流的知识工作者（研究者、写作者、学生）
- 高度重视隐私、不愿将笔记上传云端的用户
- 希望在不改变现有笔记习惯的前提下获得 AI 智能增强的用户

### 在生态中的角色
Karpathy 抽象模式在 Obsidian 生态内的**深度适配器**。它不试图取代 Obsidian 的任何能力，而是以插件形式嵌入，将 LLM 的编译与推理能力注入用户现有的笔记流，让“构建个人维基”从愿景变为可执行的本地工作流。

## 核心功能

- **知识自动提取** — 从 vault 笔记中自动识别实体（人、组织、工具、书籍、地点）、概念（理论、框架）及 9 种关系类型
- **Wiki 页面生成** — 为每个实体和概念自动生成结构化 Markdown 页面，兼容 Obsidian Bases，自动建立双向链接
- **自然语言问答** — 基于个人笔记的对话界面，每个回答附带溯源链接
- **混合搜索** — 关键词 + 语义相似度 + vault 结构三重检索，兼顾查准率与查全率
- **完全本地优先** — 默认使用 Ollama，无需注册、无需付费、数据绝不出本机
- **增量提取** — 文件保存时自动触发后台提取，无需手动操作
- **自知之明** — 当笔记内容不足以支撑回答时，明确告知用户而非胡编乱造

## 架构设计

### 整体架构概述
系统以“提取-存储-生成-查询”为核心循环，由五个关键模块构成：

1. **Extraction（提取引擎）**：遍历 vault 中的每篇笔记，调用 LLM 进行实体/概念抽取与关系识别，输出结构化数据合并到 `wiki/kb.json`
2. **KB Manager（知识库管理）**：维护 `wiki/kb.json` 作为中央知识库，支持增量更新、去重及全文索引
3. **Page Generator（页面生成器）**：从 `kb.json` 为每个实体/概念渲染 Markdown 模板，写入 `wiki/entities/` 和 `wiki/concepts/`，附带 frontmatter 和溯源双向链接
4. **Chat Engine（问答引擎）**：接收自然语言查询，并行执行关键词、语义、结构三种搜索，召回上下文后交由 LLM 生成带系统溯源引用的回答
5. **Settings & Commands（配置与命令层）**：提供命令面板入口（重新索引、单文件提取、重建 wiki）和可视化设置界面（LLM 提供商、模型选择、语言等）

所有生成内容隔离于 `wiki/` 文件夹，原始笔记永不修改，确保可用性与数据安全。

### 关键设计决策及其原因

| 决策 | 选择 | 替代方案 | 原因 |
|------|------|----------|------|
| 运行环境 | Obsidian 插件 | 独立桌面应用 / CLI | 零切换成本，直接嵌入现有笔记生态 |
| 默认 LLM | Ollama（本地） | 云 API 优先 | 隐私极优，零配置，无订阅成本 |
| 知识存储 | JSON (`kb.json`) | SQLite / 向量数据库 | 透明可审计，Git 友好，无需额外依赖 |
| 搜索策略 | 混合搜索 | 纯向量 / 纯关键词 | 笔记场景下关键词精确、语义容错、结构补全相互增强 |
| 内容隔离 | `wiki/` 文件夹 | 直接修改原笔记 | 保护用户原始数据，区分“原材料”与“加工品” |

### 数据流示例

1. 用户在 vault 中创建或修改笔记
2. 保存触发 Extraction → 逐篇调用 LLM 提取结构化信息 → 更新 `kb.json`
3. Page Generator 读取 `kb.json` → 为新增/变更的实体与概念生成/更新 Markdown 页面
4. 用户在 Chat 中提问 → 混合搜索从 `kb.json` 和 wiki 页面中召回上下文 → LLM 生成带溯源链接的回答
5. 对话历史独立保存，与笔记和 wiki 分离

## 核心机制深度剖析

### 1. 知识提取：将非结构化笔记编译为结构化图谱
插件将每篇笔记作为一个独立输入送交 LLM，使用精心调制的提示词（extraction prompt）引导模型识别：
- **5 类实体**：人物、组织、工具、书籍、地点
- **抽象概念**：思想、理论、框架等
- **9 种关系类型**：如 `uses`、`created_by`、`part_of`、`related_to`、`contradicts` 等

这种“限定类型槽”的设计在保证一致性上极其实用：它避免了 LLM 放任自流创造出难以统一处理的自定义关系，但同时限制了发现意外关联的开放性——一种为了工程可靠性而做出的刻意约束。

### 2. 知识表示：JSON 作为机器的“Schema”，Markdown 作为人的“Wiki”
不同于原始 Karpathy 设计中纯 Markdown 的方案，本实现采用双层表示：
- **`wiki/kb.json`**：存储所有实体、概念、关系及其元数据，作为系统运行的真理源。查询、生成、去重均围绕此文件展开。
- **`wiki/entities/`、`wiki/concepts/`、`wiki/sources/`**：从 JSON 渲染出的 Markdown 页面，用于人类浏览，并利用 Obsidian 的 `[[wikilink]]` 实现可视化的双向链接网络。

这一分离牺牲了“一切皆为 Markdown”的纯粹性，但换来了查询效率和增量更新的工程技术可行性。

### 3. 混合搜索：三种策略的智能加权
查询时，系统并行运行三组检索器：
- **关键词 (BM25)**：精确匹配术语，确保不会错过专门术语。
- **语义 (Embedding)**：使用本地嵌入模型（如 `nomic-embed-text`）寻找语义相近的内容，容忍不同表述。
- **结构 (Vault links)**：利用 Obsidian 现有的内链关系进行图遍历，找回结构上相关但词面上无关的笔记。

三者结果合并、去重、重排序后输入 LLM 生成回答。该设计针对个人笔记“名非恒名、意非恒意”的复杂性给出了一个务实答案。

### 4. “自知之明”：对抗幻觉的主动机制
插件在 LLM 回答前会对回答置信度进行内部评估，当可用上下文不足以支撑可靠回答时，主动声明“我的知识库中尚无足够信息”，而不是强制 LLM 生成可能失实的内容。这是对过信 LLM 输出可靠性的一种重要防御。

## 与 Karpathy 原始设计的结构化对比

| 维度 | Karpathy 原始设计 | domleca 实现 | 偏离说明 |
|------|------------------|-------------|---------|
| 原始资料 (raw) | 用户手动策展的 `raw/` 目录 | 整个 Obsidian vault 充当 raw | 降低策展门槛，但可能引入噪音 |
| Wiki 层 | 纯 Markdown 文件系统 | Markdown + `kb.json` 双重表示 | 牺牲纯粹性换取查询效率与可维护性 |
| Schema 层 | 自由文本 `schema.md`，LLM 可读可进化 | 固化在代码与插件设置中 | 可靠性升级，但失去与 LLM 共演进 Schema 的能力 |
| Ingest 操作 | 强调人机研讨，单篇摄入 | 支持全量提取和自动增量提取 | 自动化程度更高，但减少即时互动 |
| Query 操作 | 答案可回存为 Wiki 页面 | 聊天界面，回答暂未一键转为笔记 | 缺失“知识反哺”闭环（但已计划实现） |
| Lint 操作 | 主动质量审计 | 尚未实现 | 缺少知识卫生保障机制 |
| 索引 | `index.md` 及 `log.md` | `kb.json` 的内部结构 | 失去 Markdown 可读性，换得高效查询 |
| 工具链 | 强调 Obsidian、Git、CLI 工具组合 | 深度集成于 Obsidian，Git 天然可用 | 生态锁定 vs. 灵活性 |

domleca 的实现选择了一条高可落地性的路径：优先交付“提取”与“查询”两大核心价值，暂缓实现自我审计和知识回流等高级特性。这是一种务实的 MVP 策略，先让用户获得从碎片笔记到结构化百科的质变体验，再逐步完善剩余环。

## 社区反馈与工程信号（基于 feedback.md）

项目的 `feedback.md` 不仅是一份问题列表，更是其“公开建设（Building in Public）”理念的集中展现。从社区反馈中可以提炼出以下工程信号：

### 1. 最高杠杆改进：OpenAI 兼容接口
呼声最高的需求（6 位用户明确要求）是支持非 Ollama 的 LLM 后端。项目方通过 PR #6 实现了通用的 OpenAI 兼容接口，一举兼容了 LM Studio、OpenRouter、Groq、Cloudflare 等所有主流服务，而非为每个服务单独适配。这是一次极强的工程杠杆操作。

### 2. 最被认可的边界：拒绝移动端
面对移动端需求，项目方明确拒绝，理由为“在当前设备上运行本地 LLM 不切实际”。这一边界划清了产品作为桌面端深度思考工具的身份，也保护了项目资源不被分散。

### 3. 最关键的待办项：对话保存为笔记
用户指出与 LLM 的有价值对话无法沉淀回知识库，只能手动复制粘贴。项目方将此确认为“已确认差距（Confirmed gap）”，并提升为下一步优先开发的功能。这是实现 Karpathy 设计中“知识反哺”闭环的核心步骤，目前的缺失意味着知识循环尚未真正合拢。

### 4. 未竟的愿景：接入浏览器历史等外部数据
有用户提出笔记只代表所知的一小部分，建议接入浏览器历史、书签等外部信息以提升检索质量。项目方虽认可价值，但因可能与隐私优先的核心理念冲突且跨平台实现困难，决定暂不跟进。这揭示了该类工具的根本局限——知识范围被限定在用户的主动记录内。

## 适用场景与局限性

### 最适合的场景
- 个人深度知识管理：研究者、写作者、学生构建专题知识库
- 隐私敏感型用户：完全本地运行，笔记绝不上传
- 已有 Obsidian 工作流的用户：无需改变任何现有习惯即可获得 AI 增强
- 中等规模知识库：数百至数千篇笔记，此时性能与效果最佳

### 关重局限
- **平台锁定**：功能完全依赖 Obsidian，无法脱离运行
- **本地 LLM 质量天花板**：受限于用户硬件，7B 模型对复杂语义的理解可能有短板（但可通过 OpenAI 兼容接口接云端模型缓解）
- **性能瓶颈**：超大规模 vault（>10k 笔记）的全量提取耗时较长，增量策略尚待细化
- **功能子集**：原始范式中的 Lint、知识反哺等高级机制尚未实现
- **无图谱可视化**：虽然生成了结构化链接，但没有类似 Obsidian Graph View 的全局知识图谱

## 总结

`domleca/llm-wiki` 是一次成功的范式移植。它证明了“个人 AI 维基”不需要一个独立构建的重型应用，而可以是现有成熟工具的智能增强层。其核心贡献在于：
1. **用工程实践验证了 LLM Wiki 模式的产品化可行性**；
2. **提供了隐私优先、零摩擦的本地实现范例**；
3. **做出了清晰且诚实的工程取舍，最大化当前阶段的核心用户价值**。

它的局限同样诚实：知识的反哺闭环尚未合拢，自我审计能力缺位，全局可视化仍需依赖 Obsidian 原生能力。但作为个人知识管理前沿的一个开源基座，它在易用性、隐私保护和理念一致性上交出了目前最清晰的答卷。
