# Graphify 工具分析

> **PyPI**: `graphifyy` (注意双写 y)  
> **版本**: 0.7.10 | **Python**: >=3.10  
> **作者**: Safi Shamsi | [The Memory Layer 书籍](https://safishamsi.gumroad.com/l/qetvlo)

## 核心功能

### 71.5 倍 Token 压缩

Graphify 的核心价值主张。通过一次性预构建知识图谱，后续查询直接遍历图结构，无需重复将原始文件送入 LLM 上下文。

- **原理**：代码文件 → 结构化图谱节点/边（高度压缩）；查询 → 图遍历（仅涉及相关子图）
- **效果**：项目规模越大、查询越频繁，压缩收益越显著
- **局限**：初始构建需一次性消耗 Token；代码更新后需增量或全量重建

### 多模态统一入图

| 模态 | 处理方式 | 提取内容 |
|-----|---------|---------|
| 代码（23 种语言） | tree-sitter 本地解析 | AST 节点、函数/类定义、调用关系、继承关系 |
| PDF / Markdown | LLM 语义提取（用户 API 密钥） | 章节结构、关键概念、与代码的关联 |
| 图像 | Claude Vision 提取 | 架构图、UI 设计图、流程图的语义标签 |
| 视频 | faster-whisper 本地转录 | YouTube/本地视频转文本后语义提取 |
| SQL | tree-sitter-sql 解析 | 表、视图、函数、外键、FROM/JOIN 关系 |

### 自动聚类与可视化

- **Leiden 算法**：识别代码模块的社区结构，发现高内聚低耦合的模块边界
- **交互式 HTML**：生成 `graph.html`，支持缩放、筛选、社区高亮
- **分析报告**：社区规模、连接密度、跨社区依赖等统计指标

## 工作原理

```
项目文件夹
├── src/          ──→ tree-sitter ──→ 代码图谱（AST + 符号表 + 调用图）
├── docs/         ──→ LLM API   ──→ 文档语义图（实体 + 关系 + 摘要）
├── assets/       ──→ 多模态模型 ──→ 媒体资产标签图
└── README.md     ──→ LLM API   ──→ 项目级语义锚点
         │
         ▼
    图谱融合层（实体对齐、跨模态关联）
         │
         ▼
    统一知识图谱（属性图模型）
         │
         ▼
    Leiden 社区检测 ──→ 交互式可视化（graph.html）
```

### 代码解析层（本地）

tree-sitter 为每种语言生成：
- **语法树**：函数、类、变量声明的精确位置
- **符号表**：标识符的定义与引用关系
- **调用图**：函数/方法间的调用关系
- **继承图**：类与接口的继承/实现关系

### 文档语义层（LLM）

用户自行提供 API 密钥，由 LLM 提取：
- 文档中的技术概念及其与代码实体的映射
- 架构决策、设计约束等非结构化知识
- 需求与实现之间的追溯关系

### 多模态关联层

图像、音频等非文本资产通过以下方式与代码关联：
- 文件名与目录结构暗示的关联
- 文档中显式引用的图像/音频
- 视觉/听觉内容中提取的语义标签与代码实体的相似性匹配

## 使用场景

### 场景一：大规模项目入职

新成员需要快速理解百万行级代码库。传统方式：逐个文件阅读，Token 消耗巨大且效率低。Graphify 方式：预构建图谱后，通过社区检测快速识别核心模块，通过图遍历回答"这个函数的改动会影响哪些模块"。

### 场景二：AI 编程助手上下文优化

支持多平台集成：

| 平台 | 集成方式 | 安装 |
|------|---------|------|
| Claude Code | Skill | `curl -fsSL https://graphify.ai/install/claude-code \| sh` |
| Codex | Skill | 同上 |
| Cursor | MCP Server | `pip install graphifyy[mcp]` |
| VS Code | MCP Server | `pip install graphifyy[mcp]` |
| Gemini CLI | Skill | 手动配置 |
| Obsidian | 社区插件 | 插件市场 |

使用流程：
1. 构建项目图谱
2. 用户提问时，助手先查询图谱定位相关实体
3. 仅将相关子图（而非全量代码）注入上下文
4. Token 消耗降低，响应速度提升

### 场景三：遗留系统文档化

缺乏文档的老项目，通过 Graphify 自动提取代码结构并融合仅有的零散文档，生成可交互的知识图谱作为临时文档替代。

## 与同类工具对比

| 特性 | Graphify | Sourcegraph | GitHub Copilot Chat |
|-----|---------|-------------|---------------------|
| **代码图谱** | 是（AST 级） | 是（引用/定义） | 无显式图谱 |
| **多模态** | 是 | 否 | 否 |
| **查询可解释性** | 高（图遍历可追溯） | 中（搜索排名） | 低（黑箱生成） |
| **隐私** | 代码本地处理 | 云端 | 云端 |
| **实时性** | 静态预构建 | 实时索引 | 实时 |
| **Token 成本** | 预构建后极低 | 中等 | 高（全上下文） |

### 隐私模型

- **代码处理**：纯本地 tree-sitter 解析，不上传
- **文档/媒体**：通过用户自有 API 密钥调用 LLM
- **视频转录**：faster-whisper 本地模型，音频不上云
- **输出产物**：本地生成 `graph.html` 与分析报告
