### LangGraph vs LangChain Chain

- **LangChain Chain**：线性 DAG 编排，不支持循环和条件回退，适合固定流程
- **LangGraph（StateGraph）**：支持状态化、循环和条件分支，可以动态路由、回退或重复执行某个 Agent
- 条件边（conditional edges）根据当前状态选择下一跳，实现提前终止或回退重试
- `Send` API 支持多个独立 Agent 并行执行

### 面试题

> Q: 什么场景下应该用 LangGraph 而不是 LangChain Chain？

**参考答案**：  
当流程需要条件分支、循环或回退时，LangGraph 更合适。例如，当输入不符合某些条件时可以跳过中间步骤直接输出（条件路由），或当置信度不够时需要回退到上一个 Agent 重新处理（循环/回退）。LangChain Chain 适用于确定性的线性流程，如固定的数据预处理 Pipeline。

> Q: LangGraph 的 `Send` API 解决了什么问题？

**参考答案**：  
`Send` API 允许将不同的输入同时分发到多个 Agent 节点并行执行，而不是串行等待。例如，数据格式化和异常检测服务互相独立，可以并行执行以降低端到端延迟。

### 状态管理

Agent 或步骤之间如何共享状态而不互相污染：

- **共享状态（Shared State）**：通常使用字典或 TypedDict 存储全局状态，包含输入、中间结果、路由决策等
- **不可变更新模式**：每个 Agent 只能读取当前状态快照，返回一个 update 字典，由 reducer 机制（如 `operator.add` 或自定义合并函数）合并
- **命名空间隔离**：为每个 Agent 分配独立的键前缀（如 `diagnosis.result`），避免字段覆盖
- **状态回溯**：每一步的状态快照被持久化，便于调试和重放

### 面试题

> Q: 多 Agent 系统中如何保证各 Agent 之间不互相污染共享状态？

**参考答案**：  
核心策略是"不可变更新 + 命名空间"：每个 Agent 只读取当前状态的只读快照，返回增量更新。由编排框架（如 LangGraph 的 reducer）负责合并更新。同时为每个 Agent 分配独立的命名空间键（如 `agent_a.result`、`agent_b.result`），从根本上杜绝字段名冲突。

> Q: 状态回溯在 Agent 系统调试中有什么作用？

**参考答案**：  
状态回溯机制保存每一步的完整执行轨迹（输入、输出、路由决策）。当出现错误时可以：（1）定位是哪个 Agent 首次出错；（2）检查该 Agent 的上下文输入是否正确；（3）在出错节点前暂停（interrupt），人工修正状态后继续运行，形成快速调试闭环。

## 编排与路由

- **条件路由**：根据当前状态决定下一步 Agent，避免不必要的计算
- **提前终止**：简单场景跳过复杂推理步骤，直接进入处理阶段
- **并行化**：独立任务（如编码 + 合规检测）并行执行，降低端到端延迟

### 面试题

> Q: 条件路由如何降低系统延迟？

**参考答案**：  
条件路由根据输入的特征动态选择执行路径。例如，在客服系统中，一个简单问题可能只需要分类 → 回复两步，而复杂问题则需要分类 → 分析 → 生成 → 审核全流程。通过条件路由跳过不必要的步骤，可以显著降低平均处理时间。

### GraphRAG vs 传统 RAG

- **传统 RAG**：基于向量相似度检索文本块，缺乏对实体间关系的显式建模
- **GraphRAG**：在向量检索基础上融合知识图谱，建立实体间的关联（如事件-原因-结果因果链）
- 优势：
  - 通过路径推理回答多跳问题（"A 和 B 相关时，C 会怎样？"）
  - 利用图谱结构对检索结果排序和解释，降低大模型幻觉
- 实现方式：Agent 需要实体关联判断时，向图谱服务发送 SPARQL / Cypher 查询，返回的子图序列化为文本注入 LLM 上下文

### 面试题

> Q: GraphRAG 相比传统 RAG 的核心优势是什么？

**参考答案**：  
传统 RAG 只做文本级别的语义匹配，缺乏对实体间关系的显式建模。GraphRAG 融合知识图谱，（1）可以回答多跳推理问题，如"A 事件的原因有哪些，这些原因与 B 结果的关联是什么？"；（2）利用图谱结构辅助检索排序和结果解释，降低 LLM 生成时的幻觉风险。

> Q: 在 Agent 系统中如何集成 GraphRAG？

**参考答案**：  
GraphRAG 通常封装成一个工具（Tool），由相关 Agent 调用。当 Agent 的判断需要实体关系知识时，向图谱服务发送结构化查询（SPARQL 或 Cypher），返回的相关子图被序列化为自然语言文本，作为额外上下文注入 LLM 的 prompt 中。
