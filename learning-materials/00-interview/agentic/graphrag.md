# GraphRAG

GraphRAG（Graph-based Retrieval-Augmented Generation）是在传统 RAG 基础上融合知识图谱的检索增强生成范式。

---

## 一、传统 RAG 的局限

传统 RAG 的工作流程：

```
用户问题 → 向量检索（文本块相似度） → LLM 生成回答
```

存在的问题：

- **缺乏实体关系建模**：检索的是文本块级别的语义匹配，无法理解文本中实体之间的显式关系（如"药物 A 治疗疾病 B"、"疾病 B 的症状是 C"）
- **无法回答多跳问题**：对于"A 事件发生后出现 B 特征时，需要执行什么操作？"这类需要多步推理的问题，单次文本检索很难覆盖完整的推理链
- **推理链不透明**：LLM 基于拼接的上下文生成答案，检索结果之间缺乏结构化关联，大模型容易产生幻觉

---

## 二、GraphRAG 的核心思想

GraphRAG 在向量检索的基础上融合**知识图谱**，显式建模实体间的关系网络：

```
用户问题 ──→ 向量检索（文本块） ──→ 知识图谱查询（实体关系） ──→ 融合检索结果 ──→ LLM 生成
                  │                         │
                  ▼                         ▼
             文本相似度匹配              结构化查询
                                    (SPARQL / Cypher)
```

### 关键组件

| 组件 | 作用 |
|------|------|
| **知识图谱** | 存储实体（药物、疾病、症状）及其关系（治疗、禁忌、副作用）的三元组网络 |
| **向量索引** | 文本块的语义嵌入，用于初步检索 |
| **图查询引擎** | 执行结构化查询（SPARQL 或 Cypher），返回子图 |
| **融合器** | 将向量检索结果和图查询结果合并为 LLM 上下文 |

---

## 三、GraphRAG 的优势

### 1. 多跳推理

传统 RAG 只能做单跳检索，GraphRAG 可以通过图路径推理回答多跳问题：

```
问题："A事件发生后出现B特征时，需要执行什么操作？"

图谱路径：
  A ──[导致]──→ 事件D ──[表现]──→ B特征
  A ──[排除]──→ 方案C
  
检索结果：A药同时关联疾病D和禁忌药C，通过图路径链接起来
```

### 2. 结构化解释

图谱查询返回的不是散落的文本块，而是结构化的子图，可以被序列化为 LLM 更容易理解的格式（如 JSON 三元组或自然语言描述），降低大模型幻觉。

```
子图序列化示例：
"A导致事件D，事件D的常见表现包括B。同时A与C存在冲突关系。
因此，对于出现A且伴有B特征的场景，应避免采用C方案。"
```

### 3. 关系排序

图谱中的关系路径可以作为检索结果的**排序信号**——与问题实体路径连接更短的结果相关性更高，优于纯向量距离排序。

## 四、GraphRAG 的构建流程

```
                      ┌──────────────────┐
                      │   领域知识源      │
                      │ (文献/病历/指南)   │
                      └────────┬─────────┘
                               │
                    ┌──────────▼──────────┐
                    │   实体识别与抽取     │
                    │  (NER + LLM 辅助)   │
                    └──────────┬──────────┘
                               │
                    ┌──────────▼──────────┐
                    │   关系抽取与链接     │
                    │  (三元组构建)        │
                    └──────────┬──────────┘
                               │
                    ┌──────────▼──────────┐
                    │   图谱存储与索引     │
                    │ (Neo4j / Nebula)   │
                    └──────────┬──────────┘
                               │
                    ┌──────────▼──────────┐
                    │   GraphRAG 服务封装  │
                    │  (查询接口 + Tool)  │
                    └─────────────────────┘
```

### 步骤说明

1. **实体识别**：从领域文档中用 NER 模型或 LLM 抽取实体（实体名称、属性名、关系类型等）
2. **关系抽取**：识别实体间的语义关系（依赖、冲突、包含、影响等），构建三元组 `(头实体, 关系, 尾实体)`
3. **图谱存储**：存入图数据库（Neo4j / Nebula Graph / ArangoDB），建立索引
4. **服务封装**：对外提供查询接口（SPARQL / Cypher），支持根据实体名检索邻接子图

## 五、集成到 Agent 系统

GraphRAG 在 Agent 系统中通常封装为一个**工具（Tool）**，由相关 Agent 调用：

```python
class GraphRAGTool(BaseTool):
    name = "graph_rag"
    description = "基于领域知识图谱检索实体关系"

    def _run(self, query: str) -> str:
        # 1. 向量检索：找到相关实体
        entities = vector_search(query)

        # 2. 图查询：检索实体的邻接子图
        subgraph = graph_query(f"""
            MATCH (e:Entity)-[r]->(n:Entity)
            WHERE e.name IN {entities}
            RETURN e, r, n
        """)

        # 3. 子图序列化
        context = serialize_subgraph(subgraph)
        return context
```

集成到 LangGraph 的 Agent 节点中使用：

```python
def analyze_agent(state: State) -> dict:
    input_data = state["input"]
    # 调用 GraphRAG 工具获取领域知识
    context = graph_rag_tool.run(f"{symptoms} 的可能的疾病")
    # 将知识注入 LLM 推理
    diagnosis = llm.invoke(f"基于以下领域知识：{context}\n诊断：{symptoms}")
    return {"diagnosis": diagnosis}
```

## 六、GraphRAG vs 传统 RAG

| 维度 | 传统 RAG | GraphRAG |
|------|---------|----------|
| **检索对象** | 文本块（chunk） | 实体 + 关系（子图） |
| **关系建模** | 无（仅语义相似度） | 显式建模实体间关系 |
| **多跳推理** | 不支持（需多次检索） | 天然支持（图路径遍历） |
| **检索结果** | 散落的文本片段 | 结构化子图（可序列化） |
| **可解释性** | 低（只告诉"哪段文本匹配"） | 高（可展示完整推理链） |
| **构建成本** | 低（只需 embedding） | 高（需要图谱构建和维护） |
| **适用场景** | 通用问答、文档检索 | 专业领域（金融、法律、科研等） |

## 七、常见面试题

### 面试题 1：GraphRAG 相比传统 RAG 解决了什么问题？

**参考答案**：  
传统 RAG 只做文本级别的向量匹配，缺乏对实体间关系的显式建模。GraphRAG 通过知识图谱补充了结构化关系，可以：（1）回答多跳推理问题，如"A 药物和 B 症状的关联是什么？"；（2）利用图结构辅助检索排序和结果解释，降低幻觉；（3）提供可追溯的推理路径，增强回答的可信度。

### 面试题 2：GraphRAG 的构建成本高在哪里？如何降低？

**参考答案**：  
构建成本主要在三方面：（1）实体识别和关系抽取需要大量人工标注或高质量的 LLM 辅助；（2）图谱的维护更新成本高，新知识需要持续导入；（3）查询延迟比纯向量检索高。降低成本的思路：先用 LLM 自动抽取构建种子图谱，随着使用逐步迭代修正；只对高频查询的实体构建图索引，低频查询回退到传统 RAG；使用预训练的领域知识图谱（如 Wikidata、领域本体）而非从零构建。

### 面试题 3：在 Agent 系统中如何集成 GraphRAG？

**参考答案**：  
GraphRAG 通常封装成一个工具（Tool），由相关 Agent 在需要领域知识时调用。Agent 将当前问题的关键实体提取出来，向图服务发送结构化查询（SPARQL 或 Cypher），返回的子图序列化为自然语言文本后注入 LLM 的 prompt 中。在 LangGraph 中，这通常实现在 Agent 节点函数中，作为 LLM 推理前的上下文准备步骤。
